## Why this page matters

Movement history is a recurring operational requirement across payments, transfers, withdrawals, and cards.

## Endpoint pattern

`GET /movements/movements_by_types/?{type_req}=true&per_page={number}&page_num={number}`

Supported `type_req` values from Spotlight:

- `by_cobrus`
- `by_withdraws`
- `by_sends`
- `by_other_payments`

Send only one type filter at a time.

## Reconciliation checklist

<Steps>
  <Step>
    Persist Cobru webhook payloads first, then process them asynchronously.
  </Step>
  <Step>
    Reconcile a movement row with payment or withdrawal identifiers, not only by amount.
  </Step>
  <Step>
    Re-run a historical pull when a webhook fails, arrives late, or produces ambiguous state.
  </Step>
</Steps>

## Integration warnings

<Accordions>
  <Accordion title="One filter per request">
    Spotlight explicitly says not to send all type flags at the same time. Build your sync jobs to query one movement family per request.
  </Accordion>

  <Accordion title="Do not assume image fields or payer fields are stable">
    The sample response includes presentational fields such as `img` plus payer and tax fields. Treat those as informational until the current contract is revalidated.
  </Accordion>

  <Accordion title="Stable identifiers first">
    Prefer `pk`, `reference_cobru`, `url`, and operation-specific IDs over derived display fields when joining movement rows to internal records.
  </Accordion>
</Accordions>
